# Accessibility audit — Guest Experience Polish (5D-2), 2026-07

Task 9 of the experience-polish milestone. This is the committed record of a per-surface
accessibility walk of the guest client: focus order, roles/accessible names, keyboard
reachability, live-region behaviour, contrast, reduced motion, and color-only meaning — plus the
colorblind sweep and the deliberate default-theme contrast exceptions.

## Method

- Built the full stack (`npm run build`) and ran the real server (`node apps/server/dist/main.js`
  on a scratch port with a throwaway sqlite DB), then drove it **keyboard-only** with the
  Playwright MCP accessibility-snapshot capability (`browser_snapshot` reads the a11y tree;
  `browser_press_key` drives it). Surfaces reachable in a short quickstart session were walked
  live; surfaces gated behind long play (a live combat death, a hero condition, deep-floor combat)
  were audited from source and their component tests, and are disclosed as such below — following
  the Task 4 / Task 7 precedent, since no reachable content path in the current pack damages the
  hero or grants a condition quickly.
- Findings recorded per surface as the walk proceeded; each code finding fixed RED-first with a
  test; fixed surfaces re-checked.

## Baseline (pre-branch inventory, from the plan's Key facts)

The only `aria-live` region at branch start was `LogPanel` (`role="log" aria-live="polite"`).
`StatusBar` carried `role="status"`; the rest of the live/alert inventory was App-level session
banners and the two corruption alerts. Tasks 1–8 landed a large amount of a11y structure already
(every modal is `role="dialog"` + `aria-modal` + `aria-label` behind a shared focus trap; every
option list is a roving-focus `role="listbox"`; chargen steps, conclusion regions, and the hall
list are all labelled). This audit's job was to walk it, verify it, and close the gaps.

## Findings summary

- **3 findings**, all fixed (`fix: close the accessibility audit`, commit `eca8f94`).
- **1 amendment** (F2-amend, Important): the F2 fix's own code review caught that demoting
  `StatusBar` off `role="status"` had silently removed the only SR-audible path for a floor/depth
  change. Fixed in `fix: announce floor changes to screen readers` (commit `fb94a93`).
- **1 documented deliberate exception** (default-theme remembered-cell contrast).
- No focus-order or missing-label gaps were found on any walked surface — the Tasks 1–8 structure
  held up under the keyboard-only walk.

## Colorblind sweep

Every meaning that is carried by color also has a non-color channel:

| Color-carrying meaning | Non-color channel | Status |
|---|---|---|
| Condition (badge tint `--condition-color`) | `✺` glyph **+** condition name as text (StatusBar) | Pass (landed Task 7, re-verified) |
| Log/journal tone (combat/warning/system color) | Leading `⚔`/`⚠`/`⚙` glyph, and the line's own descriptive text | **Fixed (F3)** |
| Threat severity | Health **band word** as text (`wounded`, etc.); the glyph is `aria-hidden` decoration. No color-only threat coding exists in the panel or on the map (hostiles differ by glyph, not color). | Pass — no fix needed |
| Carried light (glow color) / light state | "Light: Lit/Dark" as text in the hero panel; the glow is decorative (`aria-hidden`) | Pass |
| Material palette (wall/floor/door/pillar/stair/void tints) | Each material has a distinct glyph (`#`/`.`/`+`/`O`/`<`/`>`/void) inherently | Pass (documented — glyph is the primary channel; color is supplementary) |
| Portrait / class / hall accent colors | Decorative only; the option's text label and glyph carry identity | Pass |

## Deliberate contrast exceptions (default theme)

The default "Living Tapestry" theme is a dark-fantasy palette; a few of its by-design choices sit
below WCAG AA. **The high-contrast theme is the accessible alternative and is the named
mitigation** — it re-declares every palette variable to clear AA normal-text (4.5:1) and glyph
(3:1) contrast, and that is contract-tested (`styles-contract.test.ts`, "high-contrast theme
stylesheet contract").

Measured default-theme ratios (sRGB, against the relevant background):

| Pair | Ratio | AA? |
|---|---|---|
| ink `#aab3d1` / ground `#121521` | 8.73 | ✓ |
| muted `#7d89a8` / ground | 5.21 | ✓ normal text |
| gold `#e8c879` / ground | 11.23 | ✓ |
| gold-bright `#f1d898` / panel | 10.21 | ✓ |
| alert `#db7f78` / ground | 6.34 | ✓ |
| **remembered `#4b526b` / ground** | **2.35** | **✗ — deliberate exception** |

- **Remembered cells (2.35:1).** Out-of-sight, drawn-from-memory floor is *intentionally* dim: the
  visible-vs-remembered brightness discipline requires a remembered cell to never out-shine what
  the hero's torch is illuminating right now (`cell-color.ts` / the `.cell-remembered` comment), so
  it necessarily falls below AA glyph contrast against the near-black ground. This is a mood-vs-AA
  trade the default theme makes on purpose. **Mitigation:** the high-contrast theme raises
  `--remembered` to `#8a8fa3`, which clears AA glyph contrast against its black ground (contract
  test `remembered cells clear AA glyph contrast (3:1) against ground`).

No other default-theme text pair fails AA. Material-tint glyph contrast in the default theme is
modulated by live light and is not held to a fixed AA target (the high-contrast theme's materials
are the ones held to and tested against 3:1).

## Surfaces not walked live (disclosed)

- **Conclusion screen** — reached in-engine only by a run ending; per the Task 4 precedent a live
  combat death is many turns to script reliably, so its structure was audited from
  `ConclusionScreen.tsx` (labelled regions, `role="status"`) rather than a live kill.
- **Hero-state announcer's announce-on-damage / on-condition path** — no reachable content in the
  current pack damages the hero or grants a condition in a short session (the same gap Task 7
  recorded for the condition aura/badge). The announcer's *wiring* (visually-hidden `role="status"`,
  silent on mount) and its *no-spam* property (turns advanced 0→3 with the region staying empty)
  were confirmed live; the announce-on-transition logic is covered by `hero-announce.test.ts`
  (12 cases) and `panels.test.tsx`'s `HeroStatusAnnouncer` component tests (band crossing, no
  within-band spam, condition gain).
- **Hero-state announcer's floor-transition announcements (F2-amend)** — the pinned e2e specs do
  walk a real town→depth-1 descend (`guest-play.spec.ts`), but they assert the visible
  `.status-depth` text, not the visually-hidden announcer region's spoken text; the
  `floorAnnouncement` logic itself (town↔depth wording, silent on mount/restore, silent when the
  floor is unchanged) is audited from `hero-announce.test.ts` and `panels.test.tsx`'s
  `HeroStatusAnnouncer` floor-transition cases, not confirmed against a live screen reader this
  pass.
- **The six registry overlays, trade, and house dialogs** were audited from the shared
  `OverlayScaffold` / `useDialogFocusTrap` source and their existing component tests rather than
  opened one-by-one live; they all route through one dialog+focus-trap implementation, already
  exercised by the suite.
